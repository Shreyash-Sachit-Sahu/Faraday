# Faraday

> A computer-science tutor chatbot that answers with **cited sources** — hybrid + graph RAG over a 112K-chunk corpus, generated by a **QLoRA fine-tuned Gemma 2 2B**, served behind a hardened Spring Boot API and a streaming Next.js UI. Engineered to run **end-to-end on a single 4 GB consumer GPU**.

Faraday is a from-scratch RAG system: it retrieves relevant passages from a Wikipedia/CodeAlpaca/MMLU corpus **and** a user's own uploaded documents, fuses sparse + dense + knowledge-graph signals, reranks them, and streams a grounded answer — token by token, with the exact sources it used. The retrieval stack is built by hand (no LangChain / LlamaIndex / Haystack) to own the mechanics.

---

## Highlights


- **Lifted retrieval relevance to MRR@10 0.992** (from a 0.903 BM25 baseline; P@5 0.598 → 0.711) on a **112,147-chunk** corpus **by** fusing dense embeddings + BM25 + a graph-hop leg with Reciprocal Rank Fusion and a cross-encoder reranker.
- **Sustained ~102 retrieval req/s per replica at p95 99 ms** (~5.3M requests/day, VRAM flat at 1.4 GB, **zero 5xx** across the full load ladder) **by** warm-starting every model at boot and adding a TTL cache with semaphore backpressure (`503 + Retry-After`).
- **Ran a fine-tuned Gemma 2 2B in 3.22 GB of VRAM on a 4 GB RTX 3050** (no spill) and cut completion loss **1.67 → 0.92 vs. base** **by** QLoRA-tuning a 4-bit NF4 model and streaming generations over SSE.
- **Materialized a 231K-entity / 662K-edge knowledge graph over 110K chunks (97.9% coverage)** **by** extracting entities with GLiNER and building a Neo4j co-occurrence graph, fused as a third retrieval leg for relational queries.
- **Shipped production-grade auth** — Argon2id hashing, 15-min access + 7-day **rotating refresh tokens with reuse-detection revocation**, account lockout, and per-user data isolation — **by** building a stateless Spring Security 6 chain (bearer-only, locked CORS, security headers).
- **Owned the full retrieval pipeline** (no orchestration frameworks) **by** writing every stage — clean → chunk → embed → index → fuse → rerank → graph → generate — as a standalone, `argparse`-driven module.

## Architecture

```
                        ┌──────────────────────────────────────────────┐
                        │  Next.js 16 frontend  ·  :3000               │
                        │  animated hero · auth · streaming chat · docs │
                        └───────────────────┬──────────────────────────┘
                                            │  JWT  (access in memory,
                                            │       refresh rotated on 401)
                        ┌───────────────────▼──────────────────────────┐
                        │  Spring Boot backend  ·  :8080               │
                        │  auth · users · conversations · documents     │
                        │  SSE chat proxy · @Async ingest · Flyway      │
                        └──────┬──────────────────────────┬────────────┘
                               │ JDBC                     │ HTTP / SSE
                        ┌──────▼───────┐          ┌────────▼─────────────────────┐
                        │ PostgreSQL   │          │  FastAPI AI service  ·  :8000 │
                        │ users/chats  │          │  /retrieve  /chat  /ingest    │
                        │ docs metadata│          └───┬───────────────┬──────────┘
                        └──────────────┘              │               │
                                          ┌───────────▼────┐   ┌──────▼─────────┐
                                          │ Qdrant vectors │   │ Neo4j graph    │
                                          │  + BM25 index  │   │ entities/edges │
                                          └────────────────┘   └────────────────┘
                                                      │
                                       ┌──────────────▼────────────────┐
                                       │ Gemma 2 2B + QLoRA adapter     │
                                       │ 4-bit NF4 · SSE token stream   │
                                       └────────────────────────────────┘
```

**A chat message, end to end:**

1. The browser streams a `POST /api/chat` with a `Bearer` access token (via `fetch` + `ReadableStream` — `EventSource` can't send auth headers), retrying once through token rotation on `401`.
2. The backend authenticates, persists the user message + conversation, then proxies to the AI service over SSE.
3. The AI service retrieves in parallel — **BM25** (sparse) + **dense** (Qdrant) + **graph-hop** (Neo4j) — fuses with **RRF**, reranks with a **cross-encoder**, scoped to the public corpus **plus the requesting user's own documents**.
4. It builds a prompt and **streams tokens** from the QLoRA Gemma 2 2B: `meta → sources → token… → done`.
5. The backend relays the stream live to the browser and persists the assistant message + sources on completion.

## Prerequisites

Faraday was built and validated on **Windows 11**; paths and commands below assume PowerShell. The hard constraints it was designed around:

- **GPU**: NVIDIA with **CUDA 12.1** and **≥ 4 GB VRAM** (reference: RTX 3050 4 GB). **16 GB RAM.**
- **Docker Desktop** (for the three stores)
- **Python 3.12**, **JDK 21**, **Node.js 20+**
- A **Hugging Face account** with access to the gated `google/gemma-2-2b-it` (accept its license)

> The 4 GB VRAM ceiling is a real design constraint, not a suggestion: LLMs load in 4-bit NF4 only and lazily; embedder/reranker are ≤ 150 M params; query-time NER runs on CPU so it never contends for VRAM.

## Setup

### 1. Clone & configure

```powershell
git clone https://github.com/Shreyash-Sachit-Sahu/Faraday.git
cd Faraday
Copy-Item .env.example .env
```

Fill `.env` (it is git-ignored — never commit it). At minimum set `POSTGRES_PASSWORD`, `NEO4J_PASSWORD`, and a `JWT_SECRET` (64 random bytes, base64). Generate the secret with:

```powershell
python -c "import secrets,base64;print(base64.b64encode(secrets.token_bytes(64)).decode())"
```

`GOOGLE_CLIENT_ID` is optional (email/password auth works without it).

### 2. Start the data stores

```powershell
docker compose up -d
```

Brings up Postgres (`:5432`), Qdrant (`:6333` / `:6334`) and Neo4j (`:7474` / `:7687`), each bound to `127.0.0.1`.

### 3. AI service

```powershell
py -3.12 -m venv C:\venvs\faraday
C:\venvs\faraday\Scripts\Activate.ps1

# PyTorch must come from the CUDA 12.1 index
pip install torch --index-url https://download.pytorch.org/whl/cu121
pip install -r ai-service\requirements.txt

# always set these before running pipeline / model scripts on Windows
$env:PYTHONUTF8 = "1"
$env:HF_HUB_DISABLE_SYMLINKS = "1"
```

Build the corpus and indexes **once** before retrieval works (run from `ai-service\`, venv active). Each stage is a standalone module — pass `--help` for its exact flags:

```powershell
cd ai-service
python -m app.pipeline.stage1_wikipedia      # CS-filtered Wikipedia → clean → chunk
python -m app.pipeline.stage3_codealpaca     # code Q&A corpus
python -m app.pipeline.stage4_embed_qdrant   # embed (bge-small) → Qdrant `faraday_chunks`
python -m app.pipeline.stage5_bm25           # build the BM25 index
python -m app.graph.extract_entities         # GLiNER NER (one-time GPU batch)
python -m app.graph.build_graph              # Neo4j co-occurrence graph
```

Fine-tune the tutor adapter — or point `ADAPTER_DIR` at a pre-trained one:

```powershell
python -m app.finetune.build_dataset
python -m app.finetune.train_qlora
```

Run the service (warm-starts all models, then serves):

```powershell
uvicorn app.main:app --host 127.0.0.1 --port 8000
```

Check `http://127.0.0.1:8000/health` → `{"service":"ok","qdrant":"ok","neo4j":"ok"}`.

### 4. Backend

```powershell
cd backend
.\mvnw.cmd spring-boot:run     # serves :8080; Flyway migrates Postgres on boot
```

### 5. Frontend

```powershell
cd frontend
npm install
```

Create `frontend\.env.local`:

```ini
NEXT_PUBLIC_API_BASE_URL=http://localhost:8080
NEXT_PUBLIC_GOOGLE_CLIENT_ID=        # optional, for Google sign-in
```

```powershell
npm run dev      # http://localhost:3000   (npm run build for a production build)
```

Register an account, land on `/chat`, ask a question, and upload a document to chat against it.

## API

**AI service** (`:8000`, internal — called by the backend):

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/retrieve` | Hybrid + graph retrieval (RRF + rerank) |
| `POST` | `/chat` | RAG generation, **SSE** (`meta → sources → token → done`) |
| `POST` | `/ingest` | Index an uploaded document (owner-scoped) |
| `DELETE` | `/documents` | Remove a document's vectors + graph nodes |
| `GET` | `/metrics` · `/health` | Retrieval metrics · liveness |

**Backend** (`:8080`, public API consumed by the frontend):

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/api/auth/register` · `/login` · `/google` · `/refresh` · `/logout` | Auth + token rotation |
| `GET` | `/api/users/me` | Current user |
| `POST` `GET` `DELETE` | `/api/conversations` (+ `/{id}`, `/{id}/messages`) | Conversation history |
| `POST` | `/api/chat` | SSE chat proxy (persists messages + sources) |
| `POST` `GET` `DELETE` | `/api/documents` (+ `/{id}`) | Upload, list, delete user docs |

## Results

Measured during the phased build — full detail and methodology in [`docs/PLAN.md`](docs/PLAN.md).

**Retrieval** (112,147-chunk corpus; P@5 / R@10 / MRR@10):

| Strategy | P@5 | R@10 | MRR@10 |
| --- | --- | --- | --- |
| BM25 (sparse) | 0.598 | 0.814 | 0.903 |
| Dense (bge-small) | 0.593 | 0.768 | 0.960 |
| RRF fusion | 0.655 | 0.854 | 0.965 |
| RRF + cross-encoder rerank | 0.711 | 0.912 | 0.992 |
| RRF + rerank + graph | **0.711** | **0.914** | **0.992** |

**Throughput** (async `/retrieve`, Locust): cache-hot **102.7 RPS** at 16 users, **p95 99 ms** (≈ 5.3M req/day per replica at 60% util); cache-cold 2.5 RPS, p95 470 ms; **zero 500s** across the ladder, VRAM flat at **1.4 GB**.

**Knowledge graph**: 109,815 chunks · **231,753 entities** · 662,289 `MENTIONS` · 51,879 `CO_OCCURS` · coverage **0.979**.

**Fine-tune** (Gemma 2 2B QLoRA): SFT 3,300 examples (CodeAlpaca 1,800 + Alpaca 1,500); train loss 0.862 / eval loss 0.867; **adapter completion-loss 0.921 vs. base 1.671 (Δ −0.75)**; inference **3.22 GB VRAM** (under the 4 GB ceiling, no spill).

## Design constraints & notes

- **Built from scratch, on purpose.** No LangChain / LlamaIndex / Haystack — retrieval is implemented stage by stage to learn the mechanics. Every pipeline module runs standalone (`python -m app...`).
- **Single-GPU discipline.** 4-bit NF4 weights, lazy model loading, ≤ 150 M embedder/reranker, and CPU-side query NER keep peak inference under 4 GB.
- **The backend owns truth.** Conversations, messages, sources and documents persist server-side; the UI streams and re-reads — it never persists state itself.
- **Safe rendering.** Markdown is sanitized by default (no `rehype-raw`); code is syntax-highlighted via `rehype-highlight`.
